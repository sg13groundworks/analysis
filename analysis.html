Recommended Cache-Control Strategy

Proposed HTTP caching strategy for different asset types. These headers should be applied at CDN or web server level.

Asset Type Cache-Control Header Notes
HTML documents Cache-Control: max-age=60, s-maxage=900, must-revalidate Short browser cache, 15 min CDN cache; purge on deploy.
Versioned static assets (JS, CSS, images) Cache-Control: public, max-age=31536000, immutable Long-term cache for fingerprinted files.
Unversioned static assets Cache-Control: public, max-age=86400 1 day cache while migrating to fingerprinted assets.
Fonts (self-hosted) Cache-Control: public, max-age=31536000, immutable Use font-display: swap; to avoid FOIT.
CDN-optimized images Cache-Control: public, max-age=31536000, immutable
Vary: Accept		 Allows modern formats (WebP/AVIF) based on browser support.
These values can be translated directly into CDN rules (e.g., Cloudflare/Akamai) or web server configs.

B. 

Roadmap by Tier

üî¥ Tier 1 – Critical (Sprint 1 – Immediate)

  1. Image-heavy pages (soil-types, crawl-space, r-value, problems-…, etc.)

    • Actions

      • Convert all hero & content images to WebP/AVIF.

      • Generate responsive image sets (srcset + sizes) for mobile, tablet, desktop.

      • Apply lazy-loading for below-the-fold images (loading="lazy").

      • Compress images to a target size of < 200 KB per image where possible.

    • Goal

      • Reduce image payload on these pages from 4–8 MB → < 1 MB.

  2. JS-heavy page (/locations/)

    • Actions

      • Split map/locator into a separate JS bundle loaded only when the user interacts (click “View map”, scroll into view, etc.).

      • Mark all third-party JS as async/defer.

      • Remove unused libraries (old trackers, heatmaps, A/B testing tools not in use).

    • Goal

      • Reduce JS payload from ~7 MB → < 500–800 KB.

üü† Tier 2 – High (Sprint 2)

For each glossary / resource URL in the 3–5 MB range:

  • Images

    • Batch process images through your image pipeline (Cloudflare Images / Imgix / Akamai / ImageKit, etc.).

    • Ensure max dimensions reasonable for layout (e.g., 1200px wide max, not full camera resolution).

  • CSS & JS

    • Ensure only critical CSS is inlined; rest is deferred.

    • Remove legacy JS where not needed (no maps/videos? don’t load those bundles).

üü° Tier 3 – Normal (Sprint 3)

For pages around ~400–1100 KiB:

  • Ensure:

    • HTTP/2 + compression (Gzip/Brotli) is enabled.

    • Fonts are preloaded with font-display: swap.

    • Third-party scripts are:

      • Deferred

      • Loaded through a tag manager with performance rules.

C. 

Template-Level Actions

    • Shared logic: aggressive image optimization + lazy load.

  • Location/Service-Area Template

  • Blog/Resources Template

    • Standardized hero image sizes & CDN-delivered.

2Ô∏è‚É£ Performance SRS (System Requirements Specification)

You can hand this directly to dev/infra teams.

2.1. Scope

Optimize static assets (images, JS, CSS, fonts, third-party scripts) for www.groundworks.com to meet Core Web Vitals targets.

2.2. Performance Targets

  1. LCP: < 2.5s on 75th percentile for mobile.

  2. CLS: < 0.1 on all major templates.

  3. FCP: < 1.8s on default 4G mobile profile.

  4. Total Page Weight (HTML + assets):

    • Standard pages: < 1000 KiB

    • Heavy templates (glossary/resources/locations): < 1500–2000 KiB

2.3. Functional Requirements

2.3.1. Image Optimization

  • All raster images must:

    • Be delivered in WebP/AVIF where supported.

    • Have responsive variants (srcset) and appropriate sizes attributes.

    • Use lazy loading (loading="lazy") except LCP image.

  • CMS must enforce:

    • Max upload resolution & automated compression.

2.3.2. JavaScript Optimization

  • Split JavaScript into:

    • Core bundle (critical UI).

    • Feature bundles (maps, forms, trackers) loaded conditionally.

  • All non-critical JS must be:

    • Marked defer or async.

  • Third-party scripts:

    • Loaded via a single Tag Manager with performance rules.

    • Non-essential scripts disabled on Glossary & Resource templates.

2.3.3. CSS Optimization

  • Extract and inline critical CSS per template for above-the-fold content.

  • Move non-critical CSS to:

    • Deferred stylesheets (media="print" + onload trick) or

    • Loaded after initial paint.

  • Remove unused CSS via PurgeCSS / build pipeline.

2.3.4. Caching & CDN

  • All static assets (images, JS, CSS, fonts) must:

    • Be served via CDN with immutable fingerprints in filenames.

    • Return strong cache headers (see section 4).

  • HTML pages:

    • Must not be cached with overly long TTL (e.g., 5–15 mins at CDN edge is okay with cache purge on deploy).

2.3.5. Monitoring & Alerts

  • Integrate Lighthouse CI / WebPageTest in CI/CD.

  • Track:

    • LCP, FID, CLS, TBT, page weight.

  • Set thresholds:

    • Failing builds if LCP > 3s on key templates.

2.4. Non-Functional Requirements

  • Security: No mixed content; HTTPS-only.

  • Compatibility: Modern evergreen browsers; graceful degradation for older ones.

  • Maintainability: Document templates & performance budget per template.

4Ô∏è‚É£ Recommended Exact Cache-Control Configurations

Below is a concrete configuration you can implement at CDN / web server level (NGINX/Apache/Cloudflare):

4.1. HTML Documents

  • Goal: Fresh content, but allow short edge caching.

# For HTML pages

Cache-Control: max-age=60, s-maxage=900, must-revalidate

  • max-age=60 → browser caches for 60s.

  • s-maxage=900 → CDN can cache for 15 minutes.

  • Use cache invalidation on deploy for important content changes.

4.2. Versioned Static Assets (JS, CSS, Images, Fonts with hashes)

For files like /static/js/main.abc123.js, /static/img/hero.abc123.webp:

Cache-Control: public, max-age=31536000, immutable

  • One-year cache (31536000) with immutable since filenames change on deploy.

4.3. Unversioned Static Assets (If any)

If some assets are not fingerprinted:

Cache-Control: public, max-age=86400

  • 1-day TTL. Ideally, migrate all static assets to fingerprinted paths and use the long cache from 4.2.

4.4. Third-Party Scripts

You don’t control their headers, but you control how/when you load them:

  • Always:

    • <script src="..." async></script> or <script src="..." defer></script>

    • Use a single Tag Manager and restrict tags by page-type.

  • For self-hosted wrappers/proxies (if used):

Cache-Control: public, max-age=3600

4.5. Fonts

  • Self-hosted fonts (preferred):

Cache-Control: public, max-age=31536000, immutable

  • Use:

    • font-display: swap; in CSS to avoid FOIT (flash of invisible text).

4.6. Images

  • CDN-based optimization (ImageKit/Cloudflare/etc.) should:

    • Auto-generate WebP/AVIF.

    • Attach:

Cache-Control: public, max-age=31536000, immutable

Vary: Accept

So the CDN serves optimized format by browser support.
